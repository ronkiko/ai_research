# AGENTS.md — game1

Правила разработки для game1. Общая идея проекта и стек — в `../README.md`;
архитектура и статус — в `./README.md`. **При работе в game1 приоритет у этого
файла и `README.md`, не у корневых.**

## Документирование столов

Каждый новый стол должен содержать:

1. **Литературное `RULES`** — бытовое объяснение правил (см. раздел ниже).
2. **Альтернативную постановку** в формате:
   ```
   Альтернативная постановка:
     Вопрос: <что нужно угадать>
     Правда: <условие, при котором действие верно>
     [Подсказка: <что видит агент>]
     [Это логическая функция <XOR/XNOR/...>]
   ```
3. **Правильный уровень сложности.** Стол кладётся в подпапку
   `modules/core/mechanics/levelN/`, где N выбирается по `RESEARCH_TASKS.md`.
4. **Запись в таблицу задач** в `README.md` (раздел «Какие задачи решают
   столы») — добавить строку с формулой/правилом и типом задачи.
5. **Обновление реестра.** Импорт класса и строка в `_TABLES` в
   `modules/core/casino.py`; новый уровень добавляется в список групп
   древовидного браузера в `engine.py`.
6. **Обновление `RESEARCH_TASKS.md`.** Таблица путей/уровней должна включать
   новый стол.

Если стол решает чистую булеву функцию от двух входов (как `witness`),
должна быть указана логическая формула. Если target случайный или зависит
от истории — формула может отсутствовать, но тип задачи должен быть ясен.

## Принцип: комплексная многомодульная игра, а не «всё в куче»

game1 — это набор автономных компонентов, каждый в своём файле/пакете, со
самодокументированным интерфейсом. Компоненты общаются **только через
задекларированные интерфейсы** (`modules/base.py`), никогда не лезут во
внутренности друг друга. Антипример — `~/hunter2.py`, где рендер, механика и
работа с моделью жили в одном файле и обросли костылями. Так не делаем.

### Жёсткие правила разработки

1. **Не сваливать всё в один файл.** Каждый компонент — свой файл/пакет. Новый
   стол игры — новый файл в `modules/core/mechanics/levelN/`, где N — уровень
   сложности по `RESEARCH_TASKS.md`.
2. **Компоненты общаются через интерфейсы, не через внутренности.** Движок не
   знает классов `core`; `core` не знает про сеть; стол не импортирует адаптер ИИ.
   Контракт — в `modules/base.py`.
3. **Самодокументированность.** Каждый модуль/стол имеет docstring и
   декларирует `NAME/VERSION/SUMMARY` (+ `RULES/LEARNS` для столов), доступные
   через `info()`. Описание можно получить через движок, не читая код.
4. **Движок — только загрузчик + рендер.** Игровая логика живёт в модулях. В
   `engine.py` нет правил игр и нет знания про конкретные классы столов — только
   манифест модулей и канонический `MechanicsHost`.
5. **Псевдоокна — отдельная абстракция.** Новое окно — подкласс `PseudoWindow` +
   раскладка. Движок не рисует содержимое окон сам, а передаёт окно владельцу.
   Перерисовка всегда чистая (внутренняя область заливается перед контентом).
6. **Внешний мир — через `game_api`.** Сеть, I/O, логи, всё вне игры — за
   адаптером в `game_api`. Столы и движок не делают сетевых вызовов напрямую.
7. **Движок переживает сбой компонента.** `load()` и `select_mechanics()`
   обёрнуты в try/except — статус `fail`, а не падение.

## Как взаимодействуют компоненты

```
engine.py  ──загружает──▶  modules (core, game_api)
            │                  │
            │  после загрузки ищет MechanicsHost (isinstance)
            ▼                  ▼
        CoreModule (казино) ──делегирует──▶ MechanicsAdapter
            │                                     │
            │  select_mechanics(key)               │ подключает стол (sit), делает активным
            ▼                                     ▼
        активный Mechanics (стол-среда)
            │  observe()  →  состояние
            │  step(action) → Outcome(revealed, reward, target)
            ▼
        game_api / адаптер ИИ  (агент живёт тут; стол про него не знает)
            │  act(наблюдение) → действие
            │  train(reward, target) → меняет веса (будущее)
```

- **Engine** поднимает curses, раскладывает псевдоокна, грузит модули из
  манифеста, находит `MechanicsHost` и канонически вызывает `select_mechanics()`.
- **core (казино)** — `Module` + `MechanicsHost`: домовые правила, реестр
  столов, делегирует подключение адаптеру.
- **MechanicsAdapter** — «сесть за стол»: по ключу создаёт мини-модуль,
  вызывает `sit()`, делает активным; умеет отдать описание стола по ключу.
- **Mechanics (стол)** — среда: бытовые правила (`RULES`/`LEARNS`) + `observe()`
  + `step(action)` → `Outcome(reward, target)`. Источник обучения, но не учитель.
- **game_api (адаптер ИИ)** — граница с внешним миром; здесь живёт агент/сеть
  (пока только локальный обучаемый бэкенд). Стол даёт наблюдение и исход,
  адаптер решает действие и (потом) обновляет веса.

Цикл хода (будущее, когда подключим адаптер):
`active_table.observe() → adapter.act() → active_table.step(action) →
outcome → adapter.train(outcome)`.

## Правила написания правил мини-игр (без матана)

`RULES` и `LEARNS` стола — это **бытовое объяснение для человека**, как
настольная игра непрограммисту. Никакой математики. Цель — чтобы, читая
описание, человек понял, во что играет и чему тут учится модель.

1. **Никаких формул и символов.** Не писать `σ`, `BCE`, `P(...)`, `logit`,
   `SGD`, «марковский», «кросс-энтропия», «вероятность 0.7». Само правило
   (математика) остаётся в коде как константа (`P_RIGHT = 0.7`) с коротким
   комментарием — в описание для человека оно не попадает.
2. **RULES — как объясняют настольную игру:** что за поле/предмет, что видит и
   что делает игрок, за что даётся фишка (награда), в чём «хитрость» мира —
   скрытое правило, которое игрок не знает, но может нащупать.
3. **LEARNS — что «запоминает» модель, образами и привычками, а не архитектурой.**
   Не «веса и смещение», а «одно число-привычка», «два числа: одно — насколько
   вообще любит идти направо, второе — куда склоняться, зная прошлое». Можно
   говорить «числа» и «привычка»; нельзя — «параметры», «слой», «функция потерь»
   как математику.
4. **Простые короткие предложения, бытовой лексикон.** Как если бы объяснял за
   чаем. Один абзац на RULES, один на LEARNS.
5. **Эталон — существующие столы** `ball.py` и `kormushka.py`: прочитай их
   `RULES`/`LEARNS` и пиши в том же духе. Сложность игры может быть любой, но
   тон описания — бытовым.
