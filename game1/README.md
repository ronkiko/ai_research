# game1

Первое независимое исследование в рамках `ai_research/`. Общая идея проекта и
стек — в `../README.md` (прочитай первым).

## Статус

`engine-skeleton` — движок с навбаром, интерактивными псевдоокнами выбора и
живым игровым циклом. Столы (`ball`, `kormushka`) и модели (`bias`, `logistic`)
обучаемы; режимы `supervised`, `rl` и `play` проверены прогонами. F1 открывает
единое окно игры — панель приборов: установок, статистики прогона и
литературного «Мира»; внизу полоска меню с курсором запускает/останавливает прогон,
ставит паузу и переключает темп (вплоть до режима без ограничений тиков).
Статистика (фишки, точность, шаги, веса) обновляется в реальном времени.
Анимации мира нет — для исследования нужна статистика, а не картинка.

## Архитектура (задел для структурного мышления)

Движок — это только **загрузчик + TUI-рендер**. Вся логика живёт в автономных
подключаемых модулях со самодокументированным интерфейсом. Не делаем всё в одном
файле, как в `hunter2`.

```
game1/
├── engine.py          # загрузчик + TUI: навбар, псевдоокна, контроллер панелей и прогона (run_ui)
├── config.py          # конфиг настроек (Settings): поиск/проверка/пресет/сохранение
├── ui/                # рендеринг
│   ├── theme.py       # цветовая палитра (пары curses)
│   ├── window.py      # PseudoWindow — базовое псевдоокно (бордюр/заголовок/строки)
│   ├── console.py     # ConsoleWindow — панель «Консоль»: лог движка
│   ├── navbar.py      # NavBar — всегда видимый нижний бар (F1–F4, Q)
│   ├── listpane.py    # ListPane — окно выбора: список слева + справка по курсору справа
│   ├── title_pane.py  # TitlePane — панель F1: единое окно игры (установки + статистика + «Мир» + полоска меню), прогон гоняется в ней
└── modules/           # автономные модули
    ├── base.py        # контракты: Module, MechanicsHost (+observe/step/world_lore), AiHost, ChangeLog
    ├── game_api/      # game_api — граница с внешним миром: адаптер ИИ
    │   ├── api.py        # GameApiModule(Module, AiHost): управляет моделями канонически + act/train
    │   ├── adapter.py    # AiAdapter — переключаемые модели (состояние сохраняется) + управление
    │   ├── modes.py      # режимы обучения (supervised / RL / Play) + бытовые справки к ним
    │   └── models/
    │       ├── base.py     # Model ABC: act/train/reset/params/stats (train — no-op в Play)
    │       ├── bias.py     # 1 параметр (смещение) — для игр без состояния
    │       └── logistic.py # 2 параметра (вес + смещение) — для игр с состоянием
    └── core/          # core — казино
        ├── casino.py     # CoreModule(Module, MechanicsHost): домовые правила, делегирует адаптеру
        ├── adapter.py    # MechanicsAdapter — подключает выбранный стол, драйвит observe/step/world_lore
        └── mechanics/    # столы-мини-механики (среды для обучения)
            ├── base.py     # Mechanics ABC — стол: правила + observe/step + world_lore
            ├── ball.py     # «Куда покатится мяч» — 0 состояний, 1 параметр
            └── kormushka.py  # «Кормушка» — 2 состояния, 2 параметра
```

Принципы:
- Движок ничего не знает о содержании игры, только о манифесте модулей.
- Модуль общается с движком только через интерфейс из `modules/base.py`
  (`NAME/VERSION/SUMMARY/PROVIDES`, `load() -> LoadResult`). Самодокументируем
  через docstring + `info()`.
- Псевдоокно — самостоятельная цель рендера; движок не рисует содержимое окон,
  а передаёт окна тем, кто им владеет.

### Метафора казино (core)

`core` — казино: домовые правила, выдача фишек, контроль честности (чтобы
игроки не жульничали и не подкидывали краплёные карты крупье). Столы —
выбираемые мини-механики со своими правилами (аналоги рулетки, нардов и т.п.).
«Сесть за стол» = `MechanicsAdapter.connect(key)` подключает мини-модуль и
делает его активным.

Выбор механики канонический: движок после загрузки модулей находит среди них
`MechanicsHost` (через `isinstance`) и вызывает `select_mechanics(key)` —
через интерфейс из `modules/base.py`, а не через внутренности `core`.

При старте движок применяет настройки из конфига (или пресет по умолчанию —
первые стол/модель/режим, если конфига нет), а дальнейший выбор делает оператор
через панели навбара. Применение и смены настроек логируются в Консоль.

## Интерфейс: навбар + псевдоокна

Внизу экрана всегда виден **навбар** (стиль Norton/Far):

```
F1 Старт │ F2 Models │ F3 Games │ F4 Modes │ Q Выход
```

Над навбаром — одна активная панель; F-клавиша открывает свою панель, а
повторное нажатие (или Esc) возвращает в Консоль. В окнах выбора стрелки
(↑/↓ или k/j) двигают курсор в списке, Enter — выбирает курсорный пункт.
Активный выбранный пункт помечен `•`, курсор — `▶`.

| Клавиша | Панель | Слева | Справа | Enter |
|---|---|---|---|---|
| F1 | Старт | — | единое окно игры: установоки + статистика + «Мир» + полоска меню | активирует пункт меню под курсором |
| F2 | Models | список моделей | параметры/нейроны/веса/шаги + активный режим | `select_model` |
| F3 | Games | список столов | правила игры + чему учится модель | `select_mechanics` |
| F4 | Modes | список режимов | справка по режиму (бытовая) | `set_train_mode` |

Список и справка живут в одном окне (`ListPane`, `ui/listpane.py`): справка
обновляется по курсору — то самое описание игры / справка режима. Управление
идёт через канонические контракты `MechanicsHost` / `AiHost` (из
`modules/base.py`) — панели не лезут во внутренности модулей.

Стартовая панель — Консоль (`ConsoleWindow`): лог загрузки модулей со статусом
ok/warn/fail, применения конфига и всех изменений настроек.

### Окно игры и прогон (F1)

F1 открывает **единое окно игры** — панель приборов без анимации: вверху строка
установок `Ball • Bias (Supervised)` (в Play — `(без обучения)`), двумя
столбцами — статистика прогона и литературный «Мир», внизу **полоска меню с
курсором**. Анимации мира нет: для исследования нужна статистика, псевдографика
лишь плодила сложную раскладку.

Полоска меню — вместо отдельной кнопки и строки-подсказки. По ней двигается
курсор: фон-полоска одного цвета (синий), а слот под курсором перекрашивается
в другой (циан) — видно, где курсор. Слоты на одной строке слева направо:
`Пауза · Скорость · Старт · Закрыть`; **Старт в центре**, и курсор стоит на нём
по умолчанию. `←/→` двигают курсор, `Enter` активирует подсвеченный пункт.

- **Старт** — пуск прогона (кнопка становится `■ Стоп`); повторный Enter на
  `Стоп` — стоп (остаёмся на панели, видны итоги);
- **Пауза** — пауза (тики не двигают прогон, кнопка становится `Пуск`);
- **Скорость** — цикл темпа: `1×` (300 мс/тик) → `2×` (80 мс) → `∞` (без
  ограничений тиков — `getch` неблокирующий, цикл гоняется максимально быстро);
- **Закрыть** (или `Esc`) — остановить прогон и вернуться в Консоль (в Консоль
  пишется итог: точность и число шагов).

Пока прогон идёт и не на паузе, `getch` неблокирующий — каждое срабатывание
таймера (возврат `-1`) это один тик `observe → act → step → train`, статистика
обновляется на месте. На паузе или в покое `getch` блокирующий. Во время прогона
F2–F4 игнорируются (фокус на прогоне); F1 (в Консоль), `Esc`/Закрыть и `Q` всегда
доступны. Отдельной панели прогона больше нет — цикл живёт на стартовой панели,
логика проще.

`TitlePane` имеет **фиксированный размер 76×21** — это точная usable-область над
навбаром и под заголовком при минимальном терминале 80×24 (пол движка уже
гарантирует, что она помещается). Контроллер строит панель этого размера по
центру области панелей, а внутри всё рисуется по **фиксированным
координатам-константам** — без расчёта ширин столбцов и центрирования групп.
Два столбца:

- **Статистика** — фишки (баланс: +1 за угаданную сторону, −1 за промах),
  точность по окну последние 200 ходов, число шагов и текущие веса;
- **Мир** — литературное описание «настроек мира»: как стол перекошен и
  почему чаще выпадает одна сторона. Без формул и голых чисел — живым текстом
  через `MechanicsHost.active_world_lore()`.

## Адаптер ИИ (game_api)

Переключаемые управляемые обучаемые модели. Движок управляет ими канонически
через `AiHost` (из `modules/base.py`): перечислить, выбрать (переключить),
посмотреть статистику, сбросить.

- **Переключаемость:** все модели — живые экземпляры в реестре; переключение лишь
  меняет активную, не разрушая веса и состояние прежней. Можно поучить одну,
  уйти к другой, вернуться и продолжить с места.
- **Управление:** `model_stats(key)` — нейроны, параметры, текущие веса, число
  шагов обучения; `reset_model(key)` — сброс весов к начальным.
- **Модели:** `bias` (1 параметр, без входа — для `ball`), `logistic` (2
  параметра, один вход — для `kormushka`). Пока чистый Python (прозрачные веса);
  позже сюда же подключается torch-бэкенд и Kimi через Ollama как ещё
  переключаемые модели.
- **Режимы обучения** (`modes.py`): режим — это то, как `train()` потребляет
  исход хода. Переключается через `AiHost.set_train_mode(key)`, активный —
  `active_train_mode()`. Каждый режим несёт бытовую справку (`TrainModeInfo.help`),
  которую движок печатает после выбора — чтобы оператор понимал, что выбирает.
  - `supervised` — учится «с ответом»: мир раскрывает правильное значение
    (`outcome.target`), модель тянет предсказание к нему (кросс-энтропия,
    `δ = -LR·(σ-target)·feature`). Действие — argmax предсказания. Быстро и
    устойчиво, но требует честного ответа от мира.
  - `rl` — учится по подкреплению: модель знает только награду (`outcome.reward`),
    без правильного ответа, и тянет политику к действиям, принесшим фишки (REINFORCE,
    `δ = +LR·reward·(action-σ)·feature`). Действие — сэмплирование (Бернулли):
    даёт exploration. Медленнее, но работает без «ключа ответов» — как живой
    ученик методом проб.
  - `play` — без обучения: модель играет выученными весами и не учится во время
    прогона (`train()` — no-op, веса и шаги статичны). Действие — argmax. Это
    режим «протестировать как есть»: посмотреть, как выученная модель ведёт себя
    вживую, не трогая веса.
  - На наших играх (мяч, кормушка) «предсказать мир» и «выиграть» — одно и то же,
    поэтому оба обучающих режима сходятся. Расходятся они там, где предсказание ≠
    выигрышу (будущая «контригра»).

Игровой цикл живёт в `TitlePane` (`ui/title_pane.py`) и двигается контроллером в
`engine.py` (`run_ui`): по тикам таймера `observe → act → step → train`, а в
режиме `play` шаг `train` пропускается. Статистика прогона (сумма фишек,
точность по окну, число шагов, текущие веса) считается в панели. Обучение
проверено прогонами (supervised и RL сходятся на этих играх; play веса не
меняет).

- **«Куда покатится мяч»** (`ball`) — 0 состояний, 2 действия, сеть из **1
  параметра**: мяч падает в левую/правую лунку со скрытым перекосом; агент
  нащупывает этот перекос одним числом-привычкой.
- **«Кормушка»** (`kormushka`) — 2 состояния (где корм был в прошлый раз),
  2 действия, сеть из **2 параметров** (вес + смещение): мир любит повторять
  сторону, агент учится это подмечать.

Каждый стол — это `Mechanics`-среда: `observe()` даёт состояние, `step(action)`
возвращает исход (награда + целевое значение), а `world_lore()` — литературное
описание «настроек мира» для столбца «Мир». Сам агент/сеть живёт отдельно, в адаптере
ИИ (`game_api`); движок гоняет их вместе через контракты `MechanicsHost` /
`AiHost`, не зная внутренних деталей.

## Запуск

```bash
python3 engine.py
```

Минимальный терминал 80×24, обычный интерактивный (не пайп/скрипт). Управление:
F1 — окно игры, ←/→ — курсор меню, Enter — активировать пункт (Старт пускает
прогон, Пауза/Скорость/Закрыть — понятно), Esc — назад в Консоль, F2–F4 — панели
выбора (повтор/Esc — в Консоль), ↑/↓ — курсор в списке, Enter — выбор, Q — выход
с сохранением настроек.

## Конфиг настроек (`config.py`)

При старте движок ищет `game1.conf.json` рядом с собой:
- **есть и валиден** — загружает сохранённые игру/модель/режим;
- **сломан** (нечитаемый JSON или неизвестные значения) — отклоняет с причиной
  в Консоли и генерирует пресет по умолчанию (первые игра/модель/режим);
- **нет файла** — генерирует пресет по умолчанию (один раз, дальше стартует из
  конфига).

Все эти действия логируются в Консоль через тот же `ChangeLog`, что и выборы из
панелей. Значения проверяются по каноническим хостам (`MechanicsHost`/`AiHost`),
поэтому конфиг не может выбрать несуществующую игру или модель. Изменённые в
интерфейсе настройки сохраняются в конфиг при выходе (Q). Файл в репозитории
не хранится (`.gitignore`).
